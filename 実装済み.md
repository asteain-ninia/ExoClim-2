### 設計判断（P4-6）

- **三セル構造の卓越風（地球並み固定）**: [docs/spec/02_風帯.md §4.1](docs/spec/02_風帯.md) の Pasta 標準構造（赤道収束・亜熱帯発散 30°・亜寒帯収束 60°・極発散 90°）を実装。自転速度依存のセル数変化（[docs/spec/02_風帯.md §7.1] 未確定論点）は本最小実装では固定。
- **卓越風の方向**:
    - Hadley（0–30°）: 貿易風 — 赤道向き + 西向き（順行 NH/SH）
    - Ferrel（30–60°）: 偏西風 — 極向き + 東向き
    - Polar（60–90°）: 極東風 — 赤道向き + 西向き
    - 逆行惑星では東西成分の符号が反転（`rotationSign: 1 | -1`）
- **季節移動の振幅は declination から派生**: `subtropicalShiftNH = +shift × Math.sign(declination)`、`subtropicalShiftSH = -shift × Math.sign(declination)`。NH 夏（declination > 0）に NH 亜熱帯境界が外側（35°）、SH 亜熱帯境界が内側（-25°）に動く。
- **帯状気圧パターン**: ITCZ低（-8 hPa）/ 亜熱帯高（+6）/ 亜寒帯低（-5）/ 極高（+3）。境界は |lat| 15° / 45° / 75°。
- **大陸 anomaly**: 夏半球の陸地は `-continentalPressureAnomalyHpa`、冬半球の陸地は `+continentalPressureAnomalyHpa`。`Math.sign(declination)` で判定。
- **モンスーン領域マスク**: 各経度の年内 ITCZ 中心線 min/max を事前計算し、陸地セルでその範囲内（±3° 余裕）のものを true。範囲が 5° 未満なら無効化（季節移動が小さい場所でのオーバーマッチを抑える）。
- **沿岸湧昇は最小実装で未対応**: [docs/spec/02_風帯.md §6.2] 内部派生値として将来対応。`monthlyCoastalUpwellingMask` は全 false で初期化（出力契約は満たす）。
- **ITCZ 影響帯への調整値は 0 一様**: Step 1 へのフィードバック（[docs/spec/02_風帯.md §5]）は本最小実装では未実装。`itczInfluenceAdjustmentDeg` は 0 で初期化。
- **3 つの利用者調整パラメータに集約**: 仕様 §6.1 が挙げる 6 項目（セル数・セル境界・季節移動振幅・大陸高低気圧強度・モンスーン感度・沿岸湧昇強度）から、本最小実装で実効的に効く 3 つに絞る（季節移動・大陸 anomaly・卓越風速）。残りは [docs/spec/02_風帯.md §7] 未確定論点として後続フェーズで再検討。
- **Worker Grid キャッシング機構の堅牢性**: `WorkerRunRequest` に `windBeltParams` を追加するだけで Step 2 への配線完了。Grid は引き続き Worker 側でキャッシュ、postMessage は数 KB のみ。
- **Pipeline cache の連鎖**: Step 2 のキャッシュキーに Step 1 の出力を含めることで、Step 1 がキャッシュヒット時は ITCZResult 参照同一性が保たれ Step 2 もキャッシュヒット可能（[技術方針.md §2.2.3] 部分再計算）。
- **Canvas オーバーレイ: 等間隔の風ベクトル矢印**: 経度 30° × 緯度 15° のサンプル間隔で描画（132 矢印）。風速に比例した長さ + 簡易三角の矢印先端。年平均ビューでは 12 ヶ月平均、月別ビューではその月の wind を表示。

### §6.1.1 教訓の適用方針（P4-6、[開発ガイド.md §5.1.2] AI作業開始ゲート）

> §6.1.1 IEEE 754 の負のゼロ（-0）と Vitest `toBe(0)`

P4-6 への適用:

- `prevailingWindAtLatitude` の `speedMps * (uRaw / norm)` 計算で `speedMps = 0` のとき `-0` が出うる箇所を `normalizeWind` で `if (norm === 0 || speedMps === 0) return { uMps: 0, vMps: 0 }` と特例化し、+0 を保証。
- 単体テストで `meanWindSpeedMps = 0` 縮退ケースを検証（`Math.abs(wind.uMps).toBe(0)` を使用）。
- `Math.sign(declination)` が 0 を返す春秋分のケースで `subtropicalShiftSH = -0` が生じうる。これは `cellBoundariesDeg` の値として保持されるが、後続で `+ -0` の演算で `+0` に正規化される（IEEE 754）。出力配列で `-0` が観測される可能性は低いが、cache 等価判定では Object.is で `-0 ≠ +0` となるため将来 §6 化を検討（現時点では問題なし）。

### 修正内容（P4-5b 第 2 フォローアップ）

#### 1. 凡例を fieldset/legend に統一

## 2026-04-29 Phase 4-5b フォローアップ: レイアウト縦積み + Slider 表題はみ出し修正 + ドラッグ応答性改善

P4-5b 動作確認時の利用者報告 3 件への対応:

1. **スライダー類と凡例が地図に被さる** — 2 列 grid（左:1fr / 右:360px）で、左列 (canvas 960px) が viewport 幅次第で右サイドバーに侵入していた

3. **地軸傾斜スライダーの応答が鈍い・遅延が大きい** — 各 input イベントごとに pipeline が走り、Worker への postMessage で 64,800 セルの Grid が毎回 structured clone される (~50ms × 連射 = backlog)

### 修正内容（P4-5b フォローアップ）

#### 1. レイアウト縦積み

`<aside>` を廃止し、`<section className="controls-section">` を地図セクションの**下側**に縦配置（[src/App.tsx](src/App.tsx)）:

```

[ Map (960×480, fixed)             ]
[ Season Selector                  ]
[ Legend | Orbital | PlanetBody    ]   ← 3 列 grid (240 + 1fr + 1fr)
```

CSS:

- `.app__main` を `display: flex; flex-direction: column` に変更
- `.controls-section` を 3 列 grid（240px 凡例 + 1fr 軌道 + 1fr 本体）
- レスポンシブ: ≤ 1100px で 2 列、≤ 720px で 1 列

これで地図は常に固定サイズで上部に表示され、コントロールはその下の余白を活用する（利用者提案を採用）。

#### 2. Slider 表題のはみ出し対策

`.slider__label-row` に `gap: 8px`、`.slider__label` を `flex: 1 1 auto; min-width: 0; white-space: nowrap; overflow: hidden; text-overflow: ellipsis` に変更（[src/index.css](src/index.css)）。長い表題は省略記号 `...` で切り詰められ、枠を破らない。

#### 3. ドラッグ応答性改善

##### a. Worker bridge での Grid キャッシング

Grid（180×360 = 64,800 セル × 5 フィールド）は postMessage の structured clone コストが大きく、スライダー操作で 1 入力ごとに数十 ms を消費していた。

メッセージプロトコルを 2 種類に分割（[src/worker/bridge.ts](src/worker/bridge.ts) / [src/worker/worker.ts](src/worker/worker.ts)）:

- `WorkerGridUpdateRequest`: `{ type: 'grid-update', grid }` — Grid 参照変化時のみ送信
- `WorkerRunRequest`: `{ type: 'run', id, planet, itczParams }` — 毎回送信、Grid 不要

bridge 側で `lastSentGrid` 参照と `inputs.grid` を比較し、変化時のみ `grid-update` を送る。Worker 側は `currentGrid` をキャッシュし、`run` メッセージ受信時にそれを使う。

これで通常の slider 操作（軌道・本体パラメータ変更）では数 KB の planet/itczParams のみが送られ、postMessage コストが大幅に削減される。

##### b. Connection 層の single-in-flight + replay-latest パターン

[src/store/connection.ts](src/store/connection.ts) の `runPipelineFromCurrentParams` を改修:

- パイプラインが実行中 (`isRunning = true`) なら、新しい呼び出しは `pendingDirty = true` だけ立てて即 return
- 実行完了時に `pendingDirty` が立っていれば **最新の params で再実行**（do-while ループ）
- スライダー高速ドラッグでも backlog が積まれず、最後の入力に追従する

これにより同時実行数は常に 1。drag 中に積まれた古い計算は破棄され、release 時に最新値の結果だけが反映される。

### 設計判断（P4-5b フォローアップ）

- **縦積みレイアウトを優先**: 利用者提案「地図は常に表示しつつ、下側の余ったスペースに置く」を採用。横 2 列だと viewport 幅が狭いとき canvas (960px 固定) が右サイドバーに侵入する。縦積みなら canvas 幅は常に 960px 確保され、コントロールは下に流れる。

- **Worker メッセージ型を export**: `WorkerInboundMessage` / `WorkerOutboundMessage` を bridge.ts から export し、worker.ts が import で参照。重複定義を避け、プロトコル変更時の連動を 1 ファイルに集約。

## 2026-04-29 Phase 4-5a / 4-terrain 第 2 フォローアップ: ITCZ 影響帯を一様幅表示に切替

第 1 フォローアップ（per-pixel 描画切替 + 段彩図旧版踏襲）後、利用者から「band の水平欠けがまだ稀に発生し、海岸線の形に相関しているように見える」報告。原因の追究で、第 1 フォローアップの per-pixel 化で polygon 自己交差は解消したものの、**ITCZ アルゴリズム側で band がゼロ幅まで縮退する経度が残っている** ことが判明。

### 真因（データ層の縮退）

[src/sim/01_itcz.ts](src/sim/01_itcz.ts) の `clipBandByMountains` は、各経度の影響帯範囲（中心 ±15°）に標高 > `mountainCutoffMeters` (3000m 既定) のセルが**南北両側**にある場合、`south = north = center` にクランプしてゼロ幅にする。

旧 ExoClim 移植の地形生成は緯度行ごとに高度をランクで配分するため、5 番目の標高ビン (2000-6000m) のうち約 75% が >3000m に該当し、これらは内陸の高 noise 地点に偏って分布する。結果として:

- 海上では band 連続（高地なし）
- 陸地内陸で時々ゼロ幅（高地あり）

利用者が観測した「海岸線に相関しているように見える」現象の正体は、内陸高地の分布パターンが land/ocean マスクと相関していたため。

### 修正（表示専用の一様幅化）

可視化用としては山岳切取の局所縮退は煩雑なので、**表示用 band を「中心線 ± `baseInfluenceHalfWidthDeg` の一様幅」に統一**:

- `Math.max(-90, ...)` / `Math.min(90, ...)` で緯度範囲をクランプ。
- ITCZResult 側は山岳切取済みの south/north を保持し続け、下流 Step 2/3 が消費する経路と、デバッグビュー（[要件定義書.md §2.3.5]）で個別表示する将来経路を確保。

### 仕様との整合

[docs/spec/01_ITCZ.md §4.5](docs/spec/01_ITCZ.md) の「don't cross mountains」は **シミュレーション出力レベル**の要件であり、可視化レベルでの表現を縛らない。仕様書側の更新は不要（実装済み.md と MapCanvas.tsx のコメントで対応の経緯を記録）。

## 2026-04-29 Phase 4-5a / 4-terrain フォローアップ: ITCZ 影響帯描画修正 + 段彩図を旧版踏襲

P4-5a レビュー時の利用者報告 2 件への対応。両方とも UI レイヤーの fix。

### 報告内容

1. **ITCZ 影響帯が時々水平の帯状に欠ける** — 経度（縦の罫線）方向の特定位置で帯が部分的に消える。ループ端ではない。
2. **段彩図を旧プロジェクトのものに踏襲したい**

### 1. ITCZ 影響帯の polygon pinch バグ

#### 原因

[src/ui/map/MapCanvas.tsx](src/ui/map/MapCanvas.tsx) の `drawInfluenceBand` は polygon 描画方式（north 行を西→東、south 行を東→西、closePath）だった。山岳切取（[src/sim/01_itcz.ts](src/sim/01_itcz.ts) `clipBandByMountains`）で帯の南北境界が `southLatDeg ≥ northLatDeg` に縮退するセル（特に高山が中心緯度の南北両側にある経度）では、polygon が pinch（自己交差）する。Canvas の non-zero winding rule では pinch 部の周辺がフィル対象外と判定され、横方向に欠けが発生する。

利用者観測通り「ループ端でない」のは、pinch は山岳分布に依存して任意の経度で発生するため。

#### 修正

polygon 描画から **per-pixel ImageData 描画** に切り替え:

- 新規 `buildInfluenceBandBitmap(bands)` — オフスクリーン Canvas（960×480、pan なし基準フレーム）に経度毎の独立な縦ストリップを ImageData で描画。
- 縮退セル（`northLatDeg <= southLatDeg`）は単純に skip（自己交差しない）。
- 主 Canvas には `drawImage` を 2 オフセット（norm / norm − width）で重ね描きし、経度循環を維持。

副次効果として描画コストが減少（720 vertex polygon × 3 offsets → 1 度の ImageData 構築 + 2 回の drawImage）。

`computeBandPoints` も `useMemo` で `[itcz, currentSeason]` 依存にメモ化し、bands と bitmap の二重生成を防止。

### 2. 段彩図を旧 ExoClim の配色に揃える

[../ExoClim-/components/visualizer/constants.ts] の `ELEVATION_STOPS` / `ELEVATION_COLORS` および `OCEAN_DISCRETE_COLORS` をそのまま採用:

- **陸地（discrete 5 段階）**:
    - 0-200m: `#7fb86e`（明るい緑）
    - 200-500m: `#c7db7a`（黄緑）
    - 500-1000m: `#cc9a45`（砂色）
    - 1000-2000m: `#995a32`（茶）
    - 2000m+: `#663301`（濃い焦茶）
- **海洋（discrete 3 段階）**:
    - ≥ -200m（shelf）: `#7fcdbb`
    - -4000 〜 -200m（deep）: `#1d91c0`
    - < -4000m（abyss）: `#081d58`

旧プロジェクトと同じ「物理地図」流の段彩で、高緯度の氷雪付加（`|lat| > 75°` での白ブレンド）は除去した。海氷・雪線は Step 3 海流の海氷マスクや Step 5 気温の雪線が入った段階で別レイヤーとして重ねる方針（[src/ui/map/MapCanvas.tsx](src/ui/map/MapCanvas.tsx) `cellColor` のコメントに記載）。

### 設計判断

- **per-pixel 方式の選択**: polygon を「pinch を回避する形に分割」する案も検討したが、複雑化が増す（隣接 longitude の連結性判定 + 多数の小 polygon）。per-pixel は実装も読みやすく、影響帯のような非連続性を許容する図形に適する。
- **drawImage 2 オフセット**: 影響帯ビットマップは 960×480 のフルサイズなので、`norm` と `norm - width` の 2 つで visible 領域全幅をカバーできる。3 オフセットは不要（地形ビットマップと同じ最適化）。
- **配色のソース引用**: 旧プロジェクトのコードを直接コメントで引用し、改変点（高緯度氷雪除去）も明記。再現性のあるドキュメントとして残す。

- **procedural の landFraction スケーリング**: 利用者指定 `landFraction` を地球比 `EARTH_GLOBAL_LAND_FRACTION = 0.29` で割り、各緯度の地球統計に乗じる。`Math.min(1, ...)` でクランプ。`landFraction = 0.29` でほぼ地球同等、`1.0` で陸地優勢、`0` で全海洋。
- **terrain キャッシュ in connection**: `buildTerrainGrid` は seed=0 で 64,800 セルの noise + sort で 〜100ms 級の処理。connection 層で terrain source の deepEqual 比較によりキャッシュし、軌道パラメータ等の変更では再生成しない。
- **オフスクリーン地形ビットマップ**: MapCanvas は grid 解像度（180×360）の ImageData を 1 度生成し、本 Canvas（480×960）に `drawImage` で 2 オフセット重ね描画。経度循環と pan に追従しつつ、毎フレーム再描画してもコストはほぼゼロ（ピクセル走査は grid 変化時のみ）。
- **陸海色分け**: 陸地 4 段階（低地・高原・山地・高山）+ 海洋深さ 3 段階 + 高緯度（|lat|>75°）氷化補正。`imageSmoothingEnabled = false` でセル境界を保つピクセルアート風。
- **Step 3 海流の海氷マスクと将来統合**: 現状の高緯度氷化は緯度しきい値による近似。Step 3 海流の海氷マスクが入れば置換予定（コード内に `// Step 3 海流の海氷マスクが入れば置換` コメント残置）。

### 設計判断（P4-3）

- **円軌道近似での太陽直下点緯度**: `δ(t) = -ε · cos(2π · t)`、`t = (m + 0.5) / 12`。位相慣習は「t = 0 → NH 冬至（δ = -ε）」「t = 0.5 → NH 夏至（δ = +ε）」とし、月インデックス 0 = 1 月相当を `Months12` 配列の意味と整合させる。離心率と近日点引数による南北非対称は本最小実装では未対応とし、[docs/spec/01_ITCZ.md §7.2](docs/spec/01_ITCZ.md) の論点として残す。
- **陸海補正は「夏半球の陸地割合 − 冬半球の陸地割合」**: 経度毎に算出し、`monsoonPullStrengthDeg × sign(δ) × 差分` を中心緯度に加算。NH が全陸地・SH が全海洋のとき、NH 夏（m=6）で中心線が太陽直下点より北側に振れる挙動をテストで確認。`monsoonPullStrengthDeg = 0` で陸海補正が無効化されることもテストで確認。

- **山岳横断切取**: 中心緯度から南北それぞれの方向に、影響帯の半幅以内で標高 > `mountainCutoffMeters` のセルを探し、見つかったらその直手前に境界を切り上げる／切り下げる。境界が反転（south > north）した場合は中心点の 0 幅にクランプ。山岳しきい値を 10 km まで上げると切取が無効化されることもテストで確認。
