## 1. プロジェクト概要

### 1.1 基本情報

ExoClim は、Worldbuilding Pasta が「An Apple Pie from Scratch」Part VI および Worldbuilder's Log シリーズで提示する惑星気候設計の手順を、対話型ツールとして追跡できる形で実装するものである。利用者が惑星パラメータを与えると、Pasta が示す物理ベースの導出手順を順次適用し、最終的に気候帯マップを得る。

- プロジェクト名: ExoClim
- 一次参照: Worldbuilding Pasta（一次資料の URL と版は [docs/INTENT.md](docs/INTENT.md) に集約する）
- 想定利用者: 架空世界の気候設計を行う創作者およびワールドビルディング愛好家、ならびに教育用途の利用者
- 想定利用形態: 単一利用者が手元の端末で、惑星パラメータを試行的に変更しながら気候帯マップを得る用途
- 使用環境: 未確定（[技術方針.md](技術方針.md) §1 にて Phase 3 で決定する）

本ツールが扱う対象は、主星・軌道・惑星本体・大気・海洋・地形を入力とした気候帯マップまでの導出と、その中間結果の可視化、ならびにパラメータ調整による結果比較とする。実時間天候予測、主星や地質の進化、複数惑星系の同時模擬は本ツールの対象としない。

### 1.2 システム全体像

> 技術方針確定後（Phase 3）に再点検する。本節では利用者から見た三層構成のみを定める。

ExoClim は単一利用者向けの対話型ツールとして構成し、利用者の操作と内部処理を入力・シミュレーション・表示の三層に分ける。

- 入力: 主星・軌道・惑星本体・大気・海洋・地形に関するパラメータを利用者が与える。詳細は §2.1 を参照する。
- シミュレーション: §2.2 のパイプラインを順次適用し、ITCZ（熱帯収束帯）・風帯・海流・気流詳細・気温・降水・気候帯の中間結果を順に得る。各段階の入出力契約と物理仕様は [docs/spec/](docs/spec/) に集約する。
- 表示: 各段階の中間結果と最終的な気候帯マップを、対話的に切り替え可能な形で提示する。詳細は §2.3 を参照する。永続化形式は §2.4 を参照する。

利用者はパラメータを変更したのち、シミュレーションを再実行することで結果の差分を確認できる。再計算の粒度（段階単位の部分再計算と一括再計算の選択）は §3 非機能要件および技術方針で別途定める。検証の観点と判定基準は §2.5 に従う。

## 2. 機能要件

### 2.1 入力パラメータ

利用者が直接与える入力パラメータの種類を本節で規定する。各パラメータの値域・既定値・単位・利用者向け表示語彙の詳細は §6 用語集および技術方針確定後の内部実装補遺で定める。本節では「何を利用者が触れるか」のみを述べる。

派生量（例: 光度と軌道長半径から得られる年平均放射照度）は本節の対象としない。派生計算の方針は §2.2 シミュレーション・パイプラインおよび [docs/spec/](docs/spec/) で扱う。

#### 2.1.1 主星と軌道

利用者は主星と惑星の軌道に関する以下の項目を与える。

- 主星の光度
- 軌道長半径
- 公転周期
- 軌道離心率
- 近日点引数（季節非対称性を制御する要素として用いる）

公転周期と軌道長半径・主星質量の関係（ケプラーの第三法則）に従って、いずれを基本入力とし、他を派生量とするかは技術方針で別途定める。本節では「利用者がいずれかの組み合わせで軌道を一意に指定できること」のみを要件とする。

#### 2.1.2 惑星本体

利用者は惑星本体の物理量として以下を与える。

- 半径
- 自転周期
- 地軸傾斜
- 自転方向（順行・逆行）
- 表面重力

#### 2.1.3 大気と海洋

利用者は大気と海洋に関する以下のマクロパラメータを与える。

- 表面気圧
- 温室効果の強度
- 地表アルベドおよび雲アルベド
- 気温減率（鉛直方向の温度勾配）
- 熱輸送の強度（南北および東西の混合）
- 海洋の熱容量を支配する量（海洋層の厚さ・被覆率など）

各パラメータの粒度は、Pasta 一次資料が露出している粒度を下限とする。Pasta が単一スライダーで扱う量を本ツールが過細分化することは避け、逆に Pasta が独立に扱う量を本ツールが束ねることも避ける。

#### 2.1.4 地形マップ

利用者は惑星表面の地形を以下のいずれかの方法で与える。

- プリセット選択（地球・理想化大陸など、検証用に整備された既定マップ群）
- 手続き生成（乱数種子と少数の制御パラメータによる陸海配置の自動生成）
- カスタムインポート（利用者が用意した標高・陸海データの取り込み。詳細は §2.4.1）

地形マップは陸海の二値情報に加え、標高（および海洋深度）を含む。具体的なデータ粒度（解像度・量子化）は §3 非機能要件で別途定める。

### 2.3 表示・操作

利用者が結果を観察し操作する経路を本節で規定する。具体的な配色・凡例配置・座標系の選定は技術方針確定後の内部実装補遺で扱い、本節では「利用者が見られるべきもの」と「触れるべきもの」のみを述べる。

#### 2.3.1 マップ表示

マップ表示は §2.2 の各シミュレーション段階の結果を、惑星全体の二次元投影図として提示する。

- 段階の切り替え: ITCZ・風帯・海流・気流詳細・気温・降水・気候帯のいずれを表示するかを利用者が選択できる。
- 季節の切り替え: 年内位相（少なくとも代表的な四季相当の位相）を切り替えられる。年平均表示も含む。
- 同時表示: 複数の段階を重ねて表示する場合、視認性を損なわない範囲で凡例を分離する。

#### 2.3.2 凡例・補助線

利用者の解釈を助ける補助情報を、マップ上に切り替え可能な形で提示する。

- 主要な補助線: ITCZ 中心線、風帯の境界、海流の流線および温度域。
- 凡例: 表示中のマップが何を意味するかを、単位とともに常時把握できる位置に表示する。
- 表示／非表示: 各補助情報は個別に表示・非表示を切り替えられる。

#### 2.3.3 ズーム・パン

利用者は惑星全体と部分領域を行き来しながら結果を観察できる。

- 拡大・縮小および地図のスクロール（パン）に対応する。
- 投影法の選択肢として、円筒投影と極投影など、用途に応じた切替を提供する。
- 凡例・補助線は拡大率に応じて視認性を維持する（例: 過密になる線情報の間引き）。

#### 2.3.4 グラフ表示

二次元マップ以外の情報提示として、以下のグラフを提供する。

- 緯度別プロファイル（温度・降水・風速など、年平均および季節別）
- 季節変化のタイムライン（指定地点における年内変動）
- パラメータ感度（同一地点で複数パラメータ設定を比較する図、Phase 3 以降の追加検討対象）

#### 2.3.5 デバッグビュー

シミュレーション内部の中間量を、利用者が必要に応じて追跡できる経路を提供する。

- 風帯境界決定時の探索過程（途中値・採用された境界の根拠）
- 海流の選択経路と分岐
- 各ステップで使用された中間量（旧版で UI に露出せず内部で潜在的に動いていた量を可視化対象とする）

デバッグビューは既定では非表示とし、利用者が明示的に開いた場合にのみ表示する。

#### 2.3.6 パラメータ調整UI

§2.1 で規定した入力パラメータのすべてを、本ツールの利用者インターフェースから触れられること。

- §2.1 のいずれの項目も UI 上から変更可能とする。UI 未露出のパラメータは存在しない。
- 各パラメータの操作様式（連続値・離散値・プリセット選択）は、パラメータの性質に応じて使い分ける。具体的な選定方針は §3.4 利便性で別途定める。
- 各パラメータがどのシミュレーション段階に影響するかを、利用者が UI 上で確認できること。
- パラメータ変更後の再計算契機（即時／明示的な再実行ボタン）は §3 非機能要件で別途定める。

### 2.4 入出力

外部システム・ファイルとの間で行う取り込みと書き出しを本節で規定する。具体的なデータ形式・拡張子・スキーマは技術方針確定後の §4・§5 内部実装補遺で定める。

#### 2.4.1 マップインポート

利用者は外部から地形マップを取り込んで本ツールに与えることができる。

- 取り込み対象: 標高（および海洋深度）と陸海情報を持つ二次元グリッドデータ。
- 解像度の整合: 取り込んだデータの解像度がシミュレーション側と一致しない場合、既定の補間方針で整合させる。補間方針の選択肢と既定値は §4 で定める。
- 取り込み後の編集可能性: 取り込んだマップを利用者が部分的に編集できる経路を提供する（編集 UI の詳細は §2.3 を参照）。
- 失敗時の挙動: 取り込み対象が形式違反等で読み込めない場合、利用者に原因を提示し、シミュレーション状態を破壊しないこと。

#### 2.4.2 結果エクスポート

利用者は計算結果と入力一式を外部に書き出すことができる。

- マップ画像: §2.3.1 で表示中のマップを画像として保存する。表示中の凡例・補助線の有無を選択できる。
- 数値データ: 各シミュレーション段階の数値結果（緯度・経度ごとの値）を機械可読な形式で保存する。
- 設定一式の保存と再読み込み: §2.1 のすべての入力パラメータを含む設定スナップショットを保存し、後で再読み込みすることで同一結果を再現できること。再現性の要件は §3.2 に従う。
- バージョン整合: 設定スナップショットには本ツールの版を併記し、版ずれを検出した場合は利用者に確認を求めたうえで読み込むこと。

### 2.5 検証

本ツールが提示する結果が Pasta 一次資料と整合していることを示す検証の観点を本節で規定する。具体的な許容誤差・自動比較手順・テストハーネスの構造は §4 数値検証戦略および §5 内部実装補遺で定める。

#### 2.5.1 受け入れ判定（地球パラメータでの再現性）

地球の物理パラメータを与えたとき、本ツールの出力が以下を満たすこと。

- 主要な気候帯の出現位置（赤道帯・乾燥帯・温帯・冷帯・寒帯・極帯の大まかな緯度配置）が、現実の地球と整合する。
- 大陸西岸・東岸・内陸での気候差（西岸海洋性・大陸性乾燥など）が、現実の地球と整合する。
- 季節変化（夏冬での降水パターンの反転、モンスーン的な様相など）が現実の地球と整合する。

許容誤差の定量基準は §4 で定める。本節では「定性的な特徴が再現されること」を要件とする。

#### 2.5.2 Pasta 公開図との比較

Pasta が「An Apple Pie from Scratch」および Worldbuilder's Log の各回で提示する例題（地球と特殊軌道の双方）について、本ツールに同じ入力パラメータを与えたとき、出力が Pasta の公開図と同等のパターンを示すこと。

- 比較対象: Pasta が公開している中間結果図（風帯・海流・気温・降水・気候帯）と最終気候帯図。
- 比較粒度: ピクセル一致は要件としない。気候帯の境界位置と分布傾向の一致を要件とする。
- 比較自動化: ヘッドレス実行で出力図を生成し、Pasta 公開図と並置できるハーネスを §4 で整備する。

#### 2.5.3 仮想大陸（理想化大陸）での挙動確認

実際の地球以外に、典型的な大陸配置を持つ仮想惑星でも本ツールの挙動が Pasta の説明と整合すること。

- 単一矩形大陸（赤道横断・南北半球片寄り）での西岸海洋性気候・内陸乾燥帯の出現。
- 南北分断大陸（赤道で分割された二大陸）での海流配置と気候帯。
- 全海・全陸など極端な条件での退化挙動（破綻せず、定性的に妥当な結果を返すこと）。

検証用の理想化大陸データは §4 でリポジトリ内に固定されたものを基準とし、利用者が独自の仮想大陸を比較に用いる経路は §2.4 のマップインポートを通じて提供する。

## 3. 非機能要件

機能横断で要求される性質を本章で規定する。具体的な目標値（解像度の数値・許容計算時間など）は技術方針確定後の §4・§5 内部実装補遺で確定する。本章では「何を満たすべきか」のみを述べる。

### 3.1 性能

- 解像度: 緯度経度グリッドの解像度は、Pasta 公開図と同等以上を最低基準とする。具体値は技術方針で定める。
- 計算時間: 利用者が惑星パラメータを変更しながら試行錯誤できる範囲（数秒から数十秒のオーダー）を一括再計算の目標とする。
- 部分再計算: 変更されたパラメータが特定の段階以降にしか影響しない場合は、当該段階以降のみを再計算する経路を提供すること。
- 再計算契機: 即時再計算と明示的な再実行ボタンのいずれを既定とするかは技術方針で定める。本節では、利用者が制御できる経路を確保することを要件とする。

### 3.2 信頼性

- 再現性: 同一の入力パラメータと同一の乱数種子に対し、本ツールは常に同一の結果を返すこと。
- 決定性: 並列計算・最適化・キャッシュなどの実装上の都合が、結果に観測可能な差を生じないこと。
- 数値安定性: 軌道離心率・自転周期・地軸傾斜などが極端な値を取った場合でも、明示された値域内では破綻せず、結果の意味が利用者に伝わる形で出力されること。
- 入力検証: 値域外の入力に対しては、利用者に原因を提示し、シミュレーション状態を破壊しないこと。

### 3.3 保守性

- 仕様と実装の対応: 各機能要件の根拠が要件定義書および [docs/spec/](docs/spec/) に明示されていること。実装側からも該当節を参照できる構造を維持すること。
- 教訓蓄積: 実装中に得られた失敗・落とし穴を [開発ガイド.md](開発ガイド.md) §6 に蓄積し、再発防止策として明文化すること。
- 文書役割分担: 永続文書（要件定義書・技術方針・開発ガイド・docs/spec/・docs/INTENT.md）と揮発文書（[現状.md](現状.md) / [実装済み.md](実装済み.md)）の役割分担を維持すること。
- 旧版の禁則の継承: 「動画やブログに依拠せず独自実装で済ませる」「特定モジュールを変更禁止扱いとする」「UI に露出しないパラメータを許す」など、旧 ExoClim で問題化した運用は本ツールでは行わないこと。

### 3.4 利便性

- オフライン動作: 本ツール本体は、ネットワーク接続がない環境でも完全に機能すること。一次資料の参照リンクを表示するなどネットワークを前提とする補助機能は別途独立に扱う。
- 配布: 利用者が手元の端末で本ツールを導入し起動できる形態とする。具体的な配布形態は技術方針で定める。
- 操作様式: §2.3.6 の各パラメータについて、性質に応じた操作様式（連続値・離散値・プリセット選択）を一貫した方針で割り当てること。
- 表示語彙: 利用者向けインターフェースの表示語彙は §6 用語集と整合すること。仕様本文と内部実装識別子の対応を維持すること。
