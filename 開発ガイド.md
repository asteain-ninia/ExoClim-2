# ExoClim 開発ガイド

## 重要な注意事項

> **フェーズ注記**: 以下の注意事項は主に保守・不具合修正フェーズを想定した制約である。初期実装フェーズでは「変更箇所を最小限にとどめる」「既存コメントの温存」等の制約は必要に応じて緩和される。ただし「根本原因の修正」「省略の禁止」「教訓の記録」は全フェーズで有効。

- 余計な変更は行わないこと。
- 指示されていない変更をしないこと。
- 変更箇所を最小限にとどめること。
- 既存のコメントは温存すること。
- コメントを記載するときは、本質的に必要な内容を書くこと。一時的な情報を含めないこと（「// 修正 :～」「//ここを修正」「//--修正ここから--」などの書き方をしないこと）。
- 論ずるときは、根拠となるソースコードを直接引用すること。
- コードの修正時は、「～も同様に～」といった省略は一切禁止。
- 不具合修正時は、必ず根本原因を修正すること。場当たり的な対応をしないこと。
- まずは原因の解析に注力すること。
- 実装上の失敗は本ガイドの §6 に保存していくこと。
- 実装のやり残しは現状.mdに追記していくこと。
- **文書の役割分担**:
  - `現状.md` = 未解決課題・既知不具合・残作業・テスト総数。完了した課題は原則として即時削除すること。
  - `実装済み.md` = 実装履歴・フェーズ進捗・完了した機能の記録。
  - 例外として、同一課題を複数回の作業で分担し、削除すると進行文脈が失われる場合にのみ現状.mdに一時的に残してよい。行頭に[完了]とつけ、残作業完了時に課題全体を削除する。
  - 必要な履歴や再発防止策は本ガイドの §6「開発上の注意点とよくある落とし穴」に移すこと。
- 現状.mdから記載を削除する場合、それがリストの最後でない限り欠番として残すこと。
- 要件定義書は、仕様本文（完全日本語）と内部実装補遺を分離して記述すること。
  - 仕様本文では英語識別子・内部キー名・クラス名・コード値を直接記載しないこと。
  - 実装詳細は内部実装補遺に集約すること。
- 物理仕様（各 Step のアルゴリズム）は `docs/spec/0X_*.md` に集約し、要件定義書からは参照のみ。

## 1. アーキテクチャ概要

### 1.1 技術スタック

[技術方針.md §1](技術方針.md) を参照。本ガイドは技術方針の選定結果を前提として、日々の開発で守るべき作法・規約を扱う。

### 1.2 レイヤー分割

[技術方針.md §2.1](技術方針.md) で定めた 5 層分割（ドメイン / 計算 / ワーカー / 状態 / UI）に従う。本ガイドでは特に以下の 3 点を強調する。

- **層をまたぐ依存方向は上から下への一方向に固定する**。下位層が上位層を import することは禁止。レビュー時に最優先で確認する。
- **計算層の Step は他の Step の実装を import しない**。型のみの参照とする。
- **UI 層は計算層・ワーカー層を直接 import しない**。状態層のみを参照する。

これらは旧 ExoClim の「派生値が UI に届かない」「モジュール聖域化」問題への構造的な対策である（[実装済み.md §旧 ExoClim](実装済み.md) 参照）。

## 2. 実装ガイドライン

### 2.1 一般的なコーディング規約

#### 2.1.1 言語と strict 設定

- TypeScript 5.x の strict mode を有効にする（`"strict": true` および `"noUncheckedIndexedAccess": true`）。
- ESLint と Prettier を導入し、`tsc --noEmit` と合わせて CI で検証する。

#### 2.1.2 命名規約

- 識別子は英数字とする。仕様本文で使う日本語用語との対応は [要件定義書.md §6 用語集](要件定義書.md) で管理する。
- Step 関連の型・モジュール名は [要件定義書.md §4.3](要件定義書.md) の `XxxResult` 命名に従う。
- パラメータ名は Pasta 一次資料と対応させる（例: `lapseRate` ではなく `lapseRateCelsiusPerKm` のように単位を含めるなど、§2.2.3 単位系参照）。

#### 2.1.3 コメントと一時情報

- コメントは本質的に必要な内容のみ記載する。「修正:」「ここを修正」「修正ここから」など一時的な情報を残さない（§重要な注意事項に既出）。
- TODO は GitHub Issue に外出しする。コードに残す TODO は 1 行で対応 Issue 番号を併記する。

### 2.2 物理モデル実装ガイドライン

#### 2.2.1 Pasta 一次参照の引用規約

各 Step 実装ファイル（`src/sim/0X_*.ts`）の先頭に、対応する [docs/spec/0X_*.md](docs/spec/) の節番号を引用コメントで明記する。Pasta 引用が更新された場合は、まず docs/spec/ を更新し、続いて実装側コメントを追従させる。

> 例: `// Pasta Part VIb Step 6 ITCZ サブセクション。詳細は docs/spec/01_ITCZ.md §4 を参照。`

#### 2.2.2 数値計算の決定性

- 各 Step は純粋関数として実装する（[技術方針.md §1.5.1](技術方針.md)）。
- 副作用禁止: グローバル変数の変更・現在時刻参照・乱数の種子なし呼び出しを禁止する。
- 乱数を使う Step は種子を `TIn` に含める。
- 浮動小数点数の比較は厳密一致とする。`epsilon` 比較が必要な箇所は明示的にコメントを残す。
- 並列計算（並行処理・最適化）が結果に影響を与えないことをテストで確認する（[要件定義書.md §3.2](要件定義書.md)）。

#### 2.2.3 単位系の明示

- 物理量の識別子には単位を含める（例: `temperatureCelsius`、`distanceKm`、`pressureHpa`）。
- 単位を変換する関数は専用に切り出し、変換係数の根拠を docs/spec/ から引用する。
- 角度は度（degree）で持つことを既定とし、ラジアン変換が必要な箇所のみ明示する。

#### 2.2.4 パラメータの UI 露出ポリシー

[要件定義書.md §2.3.6](要件定義書.md)「UI 未露出のパラメータは存在しない」要件を遵守する。

- パラメータを追加するときは、必ず以下の 3 セットを同時更新する。
    1. ドメイン層（`src/domain/`）の型定義
    2. 状態層（`src/store/params.ts`）への登録
    3. UI 層（`src/ui/parameters/`）への露出
- 上記 3 点を満たさないパラメータの追加はレビューで差し戻す。
- 内部派生値（例: 風帯から派生する海流入力）は `derived*` プレフィックスを付け、デバッグビュー（[要件定義書.md §2.3.5](要件定義書.md)）から確認できるようにする。

旧版で「飾りスライダー」（UI に存在するが計算に効かないパラメータ）と「隠しパラメータ」（計算に効くが UI にないパラメータ）の双方が問題化した。本規約はその直接的な対策である。

## 3. テスト戦略

### 3.1 テストの種類

[技術方針.md §1.8](技術方針.md) で定めた 3 種類のテストを以下の責務分担で運用する。

- **単体テスト（Vitest）**: 各 Step の純粋関数を入出力契約として検証。`tests/unit/sim/0X_*.test.ts`
- **契約テスト（Vitest）**: Step 間の入出力型整合性、ワーカー層と状態層の境界、永続化形式の往復（write → read で同値）。`tests/contract/`
- **E2E テスト（Playwright）**: パラメータ調整 UI からマップ表示までの一連の操作。`tests/e2e/`
- **数値検証テスト（専用ハーネス）**: 地球パラメータ・Pasta 公開図比較・仮想大陸。詳細は §4。

### 3.2 単体テストの規約

- 1 Step につき最低 1 ファイル。
- 入力: 地球パラメータと、最低 1 つの異常値（極端な離心率など）を含む。
- 出力: 値域の確認（NaN や物理的にあり得ない値を返さない）と、定性的特徴の確認（赤道で温度が極より高い等）。
- 数値検証は「許容誤差つき近似一致」と「定性的特徴」の両方を含める。前者だけだと旧版で起きた「数値は合うが分布が壊れる」事象を検出できない。

### 3.3 契約テストの規約

- 永続化形式（[要件定義書.md §4.4](要件定義書.md)）の往復: ある `Snapshot` を保存して読み込んだとき、再計算結果が元と同一であること。
- ワーカー層の部分再計算: あるパラメータを変更したとき、再計算される Step が期待通りであること（変更されないパラメータでは Step がキャッシュからスキップされること）。

### 3.4 E2E の最小セット

- 起動 → プリセット選択 → 結果表示
- パラメータ調整（スライダー操作）→ 再計算 → 結果差分の確認
- 設定スナップショット保存 → 読み込み → 同一結果の表示

## 4. 数値検証戦略

[要件定義書.md §2.5](要件定義書.md) の検証要件を、以下の 3 ステージのハーネスで満たす。

### 4.1 地球パラメータ再現性

- 入力: `tests/fixtures/earth.json`（[要件定義書.md §2.5.1](要件定義書.md) の「地球パラメータ」）
- 検証対象:
    - 主要気候帯の出現位置（赤道帯・乾燥帯・温帯・冷帯・寒帯・極帯の緯度配置）
    - 大陸西岸・東岸・内陸の気候差（西岸海洋性 / 大陸性乾燥）
    - 季節変化（夏冬での降水反転・モンスーン的様相）
- 許容誤差: 気候帯の境界位置で ±5° 緯度、面積比で ±20%（Phase 4 実装着手後に再点検）。
- 失敗時の挙動: 数値だけでなく、地球の Köppen 分類との並置画像レポートを出力する。

### 4.2 Pasta 公開図比較

- 入力: Pasta が「An Apple Pie from Scratch」および Worldbuilder's Log で示す例題セット
- 比較対象: Pasta 公開の中間結果図（風帯・海流・気温・降水・気候帯）
- 比較粒度: ピクセル一致は要件としない。気候帯の境界位置と分布傾向の一致を要件とする（[要件定義書.md §2.5.2](要件定義書.md)）。
- 運用: Pasta 公開図そのものはリポジトリに含めず、利用者が `tests/fixtures-external/` に配置する。欠落時はテストをスキップして CI を通す（[技術方針.md §2.3.3](技術方針.md)）。

### 4.3 仮想大陸

- 入力: `tests/fixtures/idealized_continent_*.json`（典型的な大陸配置を持つ仮想惑星、[要件定義書.md §2.5.3](要件定義書.md)）
- 検証対象:
    - 単一矩形大陸（赤道横断・南北半球片寄り）での西岸海洋性気候・内陸乾燥帯の出現
    - 南北分断大陸（赤道で分割）での海流配置と気候帯
    - 全海・全陸など極端条件での退化挙動（破綻なし）
- 失敗時の挙動: 該当大陸配置の出力マップと、期待される定性的特徴のチェックリストを並置する。

### 4.4 数値検証の運用ルール

- 上記 3 ステージは CI 必須とする（外部資産依存の §4.2 は欠落時スキップ可）。
- ハーネス自体は決定性を要する（同入力 → 同出力で常に同じ判定）。
- 検証失敗時は実装の修正だけでなく、対応する [docs/spec/](docs/spec/) の §7 未確定論点を見直し、必要なら一次資料に立ち戻る。

## 6. 開発上の注意点とよくある落とし穴（教訓カタログ）

> **方針**: 実装で遭遇した失敗・落とし穴・再発防止策をここに蓄積する。`実装済み.md` への履歴記録だけでは再発防止扱いとしない。本章への教訓化が必須。
> 
> **書式**: §6.X.Y の階層で番号付け。各教訓は「現象」「原因」「再発防止策」の3点を明記。

### 6.0 教訓の使い方と更新ルール

- 作業着手前に §6 全体を一読し、関連する節を §5.1.2 ゲートで引用すること。
- 新規教訓を追加するときは、既存の節と被らないか確認すること。
- 旧 ExoClim から持ち越したい教訓があれば、再現性を確認したうえで本章に明文化する。**旧コードベースの感想・愚痴ではなく、新コードベースで再発させないための具体的な禁則・確認手順として書く。**

### 6.1 数値計算の落とし穴

#### 6.1.1 IEEE 754 の負のゼロ（-0）と Vitest `toBe(0)`

- **現象**: `solarDeclinationDeg(m, 0)` のように物理係数を 0 に縮退させたとき、`-axialTiltDeg * Math.cos(...)` が `-0` を返した。`expect(value).toBe(0)` は `-0` を `0` と区別する（Vitest は `Object.is` 比較）ため、論理的には等しいのに失敗する。
- **原因**: IEEE 754 浮動小数点の符号付きゼロ。負の数 × 正の数 = -0 のような式で発生する。
- **再発防止策**:
    - 物理量の 0 状態を assert するときは `expect(Math.abs(value)).toBe(0)` または `expect(value).toBeCloseTo(0, n)` を使う。
    - 縮退ケース（軸傾斜 = 0、離心率 = 0、海洋被覆率 = 0 など）のテストで `-0` が出る計算式を含むかを事前に確認する。
    - 該当する計算式: `-x * cos(...)` / `-y * sin(...)` など、定数係数が負の積。

#### 6.1.2 線形 insolation→温度マッピングの破綻（polar day での過剰加熱）

- **現象**: Step 5 気温を「`temp = baseline + (insolation - mean) × scale` の単一線形式」で実装したところ、緯度 85° NH 7 月（polar day）で `dailyInsolationFactor = 0.40` が赤道夏 `0.32` を上回り、温度が +25°C 程度まで上がった。実 Earth は同位置で 0–5°C。これに伴い `summerMax > 0 °C` で雪氷判定が外れ、極帯が雪氷から消えてしまった。
- **原因**: 実大気では Stefan-Boltzmann 則 (T ∝ Q^¼) に加え、雪氷の融解潜熱・低水蒸気・高アルベドで polar day insolation の余剰がほぼ温度に変換されない。線形変換ではこれを再現できない。
- **再発防止策**:
    - 緯度別ベース温度は **年平均と季節振幅を別スケール** で扱う（年平均 ≈ 350°C/規格化日射、季節 ≈ 80°C/規格化日射）。年平均スケールが大きいほど緯度勾配が出て、季節スケールが小さいほど polar day 暴走を抑制できる。
    - `INSOLATION_TO_TEMPERATURE_SCALE` のような単一定数を導入する前に、極夏／赤道夏／極冬／赤道冬の 4 隅で実 Earth に近い値が出るか手計算で検証する。
    - 該当実装: `src/sim/05_temperature.ts` `computeLatitudeBaseTemperatureCelsius`。

#### 6.1.3 雪氷フィードバック反復で同一セルを多重冷却

- **現象**: 反復ループ内で「累積雪氷マスク `snowIceMask` のセル全部に固定冷却 `−5°C` を加算」していたため、N 回反復で既存 ice セルが N×5°C 冷却され、極端に温度が下がる現象が発生した（ループ収束よりも単調冷却）。
- **原因**: フィードバックは「新規に氷面に変わったセル」に対してのみ albedo 上昇 → 冷却を適用すべき。既存 ice はもう冷却済み。実装でこの区別がなく、累積マスクをそのまま冷却対象にしていた。
- **再発防止策**:
    - 反復の各ラウンドで「`summerMax ≤ 0` かつ累積マスク未登録」のセルを抽出し、それにのみ冷却を適用する。
    - 反復が収束する（新規 ice ゼロ）になったら break する。
    - 該当実装: `src/sim/05_temperature.ts` `computeTemperature` 内の「7. 雪氷アルベドフィードバック」ループ。

### 6.2 テスト・E2E の落とし穴

#### 6.2.2 経度循環マップの cull マージンと「単一オフセット描画」の落とし穴

- **現象**: ユーザが地図を横方向にパンしたとき、(a) 等温線の温度ラベルが特定の pan 位置で完全に消失する、(b) 風矢印が東縁付近で部分的に消える、という症状が発生した。
- **原因**:
    - **(a) ラベル**: 等温線本体は 3 オフセット（`norm-W / norm / norm+W`）でループ描画していたが、ラベルだけ単一オフセット `norm` で描いていた。pan で経度 0° 付近のタイルが画面外に出ると、ラベルもそのまま画面外に流れて消える。
    - **(b) 矢印**: cull check `x0 < -50 || x0 > CANVAS_WIDTH_PX + 50` のマージンが、サンプル間隔（30° = 約 105px）の半分（52.5px）より小さく設定されていた。3 タイル描画でも、タイル境界の隣接サンプルが両側のタイルで culling され、東西の地図端付近に空白ができる（pan の値次第で空白の位置が動く）。
- **再発防止策**:
    - **オーバーレイ要素は必ず 3 オフセットで描く**: 線、矢印、ラベル、マーカーなどタイル単位で循環表示する要素は、`[normPanPx - CANVAS_WIDTH_PX, normPanPx, normPanPx + CANVAS_WIDTH_PX]` の 3 オフセットすべてで描画する。例外を作らない（特にラベル単独の単一オフセット描画は要注意）。
    - **離散サンプル要素（風矢印など）は cull を行わず Canvas クリッピングに任せる**: 矢印数が小さい場合（396 程度）はパフォーマンス影響がゼロのため、明示 cull を入れない。cull check を入れると「サンプル間隔と margin の関係」で wrap 矢印が東縁の外側ぎりぎりで消えるバグが入りやすい。pan によって「東縁を完全に超える瞬間まで wrap 矢印が現れない」という症状が出る。
    - **どうしても cull が必要なときは margin ≥ サンプル間隔（半分ではダメ）**: 高密度サンプル（500以上の矢印など）でパフォーマンス問題が出たときに限り cull を入れる。マージンはサンプル間隔の **半分ではなく丸ごと** + 描画要素サイズを取らないと、隣接タイルの wrap が culling される。
    - **連続線（segment list）は両端 cull で十分**: ITCZ centerline や等温線本体のように beginPath/lineTo/stroke で繋ぐ線は、Canvas のクリッピングが効くため両端が画面外（`x0 > W+50 && x1 > W+50`）のときだけ skip すれば良い。

### 6.3 UI が計算層出力を実際に消費しているかの確認

#### 6.3.1 計算結果が UI 描画に反映されない（独自実装の取り残し）

- **現象**: P4-15 で `OceanCurrentResult.monthlySeaIceMask` に「寒流沿い東岸海氷延長」を加算したが、UI には反映されなかった。原因は `buildSeaIceBitmap` ([src/ui/map/MapCanvas.tsx]) が `oceanCurrent` を一切参照せず、`grid` と `seaIceLatitudeThresholdDeg`（緯度しきい値）のみで二値判定して描画していたこと。モデル更新と UI 更新が独立に発生し、UI 側が古い独自実装を維持していた。
- **原因**:
    - 計算層パラメータの UI 露出（[§2.2.4]）は「ドメイン型 / store / UI スライダー」の 3 点で確認するルールがあるが、**計算層 _出力_** （mask / グリッドマップ / 派生量）が UI に届くかは同様の確認手順がなかった。
    - P4-15 完了報告では「ユニットテストで mask 計算が正しい」「E2E で `legend-sea-ice` トグルが動く」を確認したが、「実際に追加された延長セルが画面に表示されているか」は手動確認していなかった。
- **再発防止策**:
    - **計算層出力を変更したら、UI 側のビットマップ生成 / 描画関数が当該出力を参照しているかを grep で確認する**。例: `monthlySeaIceMask` を変更したら `grep monthlySeaIceMask src/ui/`。
    - **完了報告に「手動動作確認」を含めるルール（[指示.md] フェーズ別指示で既定）を、視覚変化を伴う実装では特に厳守**。スクリーンショット or 「○月で東岸に新たな白覆いが見える」など具体的な観測点を含める。
    - **UI 描画関数の引数を見直す**: `buildSeaIceBitmap(grid, threshold)` のように「派生量だけで完結する」シグネチャは「計算層出力を参照しないこと」を構造的に強制してしまう。出力を参照すべきなら mask を引数に取る形（`buildSeaIceBitmap(grid, mask, threshold)`）が望ましい。新規 UI 関数を追加するときは、計算層出力をどう参照するかをまず設計する。
    - 該当実装: P4-16 で `buildSeaIceBitmap` を `(grid, mask, threshold)` シグネチャに修正、月別表示時は実マスクを参照するように変更（[src/ui/map/MapCanvas.tsx]）。

#### 6.3.2 Canvas 指紋テスト（粗いサンプリング）の感度低下

- **現象**: P4-17 で polar gyre streamline + collision point マーカーを precipitation overlay の上に描画したところ、`tests/e2e/p4-10-precipitation.spec.ts` の「ITCZ 影響帯半幅を 0 にすると Canvas 描画が変わる」テストが失敗。`canvasFingerprint` が `slider 15 → 5`（クランプ後）の変更で全く同じ値（20941753）を返した。
- **原因**:
    - `canvasFingerprint` は 16 ピクセル毎にサンプリングして R+G+B の総和を取る軽量実装。粗いサンプリングのため、変更領域が小さい / 重なる新規描画に隠れる場合に差分を検出できない。
    - P4-17 で追加した streamline / collision point は precipitation overlay の上に重畳描画される。very_wet → wet の色差（青寄り → 水色寄り、ΔRGB ≈ 140）よりも、streamline (light blue 'a0c8e0') / collision point (yellow/purple) で被覆される領域が増えると、サンプリング点の見かけ上の総和差分が縮小する。
    - スライダーが `min={5}` で値 0 設定が 5 にクランプされる仕様のため、実際の変更幅は 15 → 5（10 度縮小）と思いのほか小さく、初めから検出余裕がなかった。
- **再発防止策**:
    - **新規 overlay を既存 overlay の上に重畳追加するときは、既存の Canvas 指紋テストの感度を再点検する**。重畳が増えるほど、下層の差分が drowned out されて検出限界が下がる。
    - **fingerprint テストは「明確に大きな差分が出る方向」のスライダー操作で書く**。例: 15 → 5（10 度縮小、影響範囲狭まる）よりも、15 → 30（15 度拡大、影響範囲が倍に広がる）を選ぶと差分が顕著になる。
    - **将来的には specific-pixel チェック（`getImageData(x,y,1,1)`）も併用**。検出領域が小さい場合でも特定ピクセルの色を厳密比較できる（[§6.2.1] にも同趣旨の記述あり）。
    - **テスト名と実挙動の乖離に注意**。「0 にすると」と書いても min クランプで 5 になる場合は、テスト名で実値（または相対操作）を表現する。
    - 該当実装: P4-17 で `tests/e2e/p4-10-precipitation.spec.ts:98` を「最大に上げる（15 → 30）」に変更し、明確な差分を出すよう修正。

#### 6.2.1 Step 追加で初期パイプライン完了時間が伸び、E2E 待機が不足する

- **現象**: P4-9 で Step 5 気温を pipeline に追加したところ、P4-6（Step 2 風帯）の「卓越風トグル ON で Canvas が変わる」テストや P4-5c の「monsoonPull=0 で年平均 ITCZ が直線化」テストが失敗。Canvas 指紋が `before == after` で同一値となり、トグル/スライダー操作の結果が反映されていなかった。
- **原因**: dev mode の Worker 上で動く pipeline の初回実行 + 個別パラメータ変更後の再計算が Step 5 追加で長くなった。`beforeEach` で 200ms 待機していたものが Step 5 後は不足し、`windField` / `temperature` 等が `null` のまま測定が走っていた。
- **再発防止策**:
    - 新しい Step を pipeline に組み込んだら、既存 E2E の `beforeEach` 待機（`waitForTimeout`）が新 pipeline の所要時間を吸収できるか必ず確認する。Step 増加時の経験的目安として `beforeEach` ≥ 900ms、パラメータ変更後の再描画待機 ≥ 900ms。
    - 待機を伸ばすのではなく「特定セルが特定色になる」ような `waitForFunction` で完了を判定するほうが堅牢。次の Step 追加時に検討する（[要件定義書.md §3.4] テスト規約）。
    - Canvas 指紋ベースの差分検出はサンプリングが粗い（16 ピクセル毎）ため、わずかな線描画の移動を捕えられないことがある。重要な視覚変化テストは別途 specific-pixel チェックも併用する。
