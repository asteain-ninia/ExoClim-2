# ExoClim 技術方針

> **状態**: Phase 3 で決定。本文は未確定。

## 0. 決定方針

- 旧 ExoClim（React 19 + Vite 6 + TypeScript 5.8 + D3 + Recharts）の構成は**前提としない**。再検討から始める。
- 採用技術は要件（オフライン動作・地図描画・パラメータ操作UI・ヘッドレス出力可能性）と AI 協働開発の容易さから選定する。
- 各技術選定には「なぜそれを選んだか」「他の候補に対する優位性」を明記する。

## 1. 技術スタック

旧 ExoClim の構造的問題（[実装済み.md §旧 ExoClim](実装済み.md) 参照）は platform 選定ではなく governance / design に起因していた（UI 未露出パラメータ・モジュール聖域化・Pasta 準拠の薄さ）。platform 選定では同等構成も評価対象に含めたうえで、AI 協働の容易さと旧版改善のしやすさで再決定する。

### 1.1 デスクトップ vs Web

**選定**: 静的 Web SPA を一次提供形態とする。Tauri 2 によるデスクトップ単一バイナリ化を将来オプションとする。

**理由**:

- [要件定義書.md §3.4](要件定義書.md) の利便性要件: 「利用者が手元の端末で導入・起動できる形態」「オフライン動作」を満たす。
- 静的 SPA は Service Worker + IndexedDB で完全オフライン動作可能。
- 配布は zip / GitHub Pages / 単一 HTML のいずれも選択可能で、利用者の環境を問わない。
- AI 協働における学習データ量が最大（後段の §1.3 と整合）。

**候補比較**:

| 候補 | 採否 | 評価 |
| --- | --- | --- |
| 静的 Web SPA | **採用** | 開発が単純、AI 協働容易、配布柔軟、PWA でオフライン対応可 |
| Tauri 2（デスクトップ単一バイナリ） | 将来オプション | ファイルシステムアクセスが安定し配布が単一バイナリで済むが、Rust 工程の追加コストと AI 協働のデータ量で静的 Web に劣る。Phase 4 後半で再検討 |
| Electron | 不採用 | バンドルサイズと性能で Tauri に劣り、Tauri が選択肢にある以上選ぶ理由が薄い |
| Native（Qt / SwiftUI 等） | 不採用 | 対象 OS 依存が増え、AI 協働で不利。本ツールの規模に対して過剰 |

**リスクと対応**:

- File System Access API は Firefox 等で未対応。`<input type="file">` + ダウンロード経路を fallback として §2.2 で規定する。
- 静的 SPA は「インストール感」が薄く、利用者が起動方法に戸惑う可能性がある。配布時のドキュメントで明示する（[開発ガイド.md](開発ガイド.md) §3 で扱う）。

### 1.2 言語

**選定**: TypeScript 5.x（strict mode）

**理由**:

- [要件定義書.md §3.3](要件定義書.md) 保守性「仕様と実装の対応」を、ステップ間の型契約として機械的に検証できる。
- AI 協働における型推論の文脈ヒントが得られる。
- 旧版でも採用されており、移行コストを増やさず governance 改善に集中できる。

**候補比較**:

| 候補 | 採否 | 評価 |
| --- | --- | --- |
| TypeScript | **採用** | 型による契約・AI 協働支援・エコシステムの三拍子 |
| JavaScript（型なし） | 不採用 | Step 間の入出力契約が緩み、旧版で発生した「派生値が UI に届かない」型の問題を防げない |
| ReScript / Elm | 不採用 | 型系統は強いが AI 協働の学習データ量が少なく、本プロジェクトの目的（Pasta 手順の追跡）に対して整備コストが過剰 |
| Rust（all in） | 不採用 | UI 層には別途 web-tech が必要で、純 Rust 単独では構築が困難 |

### 1.3 UIフレームワーク

**選定**: React 19

**理由**:

- AI 協働の学習データ量が最も多く、コードレビュー・補完・refactoring の支援が最大。
- エコシステム（コンポーネントライブラリ・型定義・開発ツール）の成熟度。
- 旧版採用実績あり。旧版の問題は React に起因するものではなく、状態管理（過剰な Context 依存）と可視化（D3 と React の不整合）に起因していたため、本仕様ではこれらを §1.4 / §1.6 で改善する。

**候補比較**:

| 候補 | 採否 | 評価 |
| --- | --- | --- |
| React 19 | **採用** | AI 協働支援・エコシステム成熟度ともに最大 |
| Svelte 5 | 不採用 | reactive primitives でコード量は少なくなる魅力があるが、AI 協働の学習データ量で React に劣る。本プロジェクトの最重要要件（AI 協働での Pasta 追跡）を優先する |
| Vue 3 | 不採用 | React と同等の支援だが、エコシステム選択肢が React より薄い |
| Solid | 不採用 | React 風 reactive で軽量だが、学習データ量で劣る |

**リスクと対応**:

- パラメータスライダー多数による再描画パフォーマンス → §1.4 状態管理（Zustand）と §1.5 計算層（Worker 分離）で緩和。
- React の useEffect 多用による副作用の見通し悪化 → §2.1 レイヤー分割で純粋計算層を React から完全分離する。

### 1.4 状態管理

**選定**: Zustand

**理由**:

- [要件定義書.md §3.1](要件定義書.md) 性能要件「部分再計算」を、Step ごとの store 分割で表現しやすい。
- React Context のみより数桁速い再描画性能（パラメータスライダーで顕著）。
- ボイラープレートが最小で、AI 協働での編集・差分レビューが容易。

**候補比較**:

| 候補 | 採否 | 評価 |
| --- | --- | --- |
| Zustand | **採用** | 軽量・高速・型安全・AI 協働容易 |
| Jotai（atom ベース） | 不採用 | 部分更新は強いが、Step 間の集約（pipeline 全体の再計算）が atom 関係図として複雑になる |
| Redux Toolkit | 不採用 | 本ツール規模に対して過剰、ボイラープレート増加 |
| React Context のみ | 不採用 | パラメータ変更で全体再描画、性能不足 |

### 1.5 計算層

**選定**: TypeScript 純粋関数として実装し、Web Worker 上で実行する。

**理由**:

- [要件定義書.md §3.2](要件定義書.md) 信頼性「再現性」「決定性」: 純粋関数なら同入力 → 同出力が機械的に保証される。
- [要件定義書.md §3.1](要件定義書.md) 性能「部分再計算」: 各 Step を独立モジュール化し、変更パラメータが影響する Step 以降のみ再実行する経路を持たせやすい。
- UI スレッドのブロッキング回避（パラメータ変更時の応答性）。
- AI 協働で実装と検証が容易（型・テスト・差分レビューがすべて TypeScript で完結）。

**候補比較**:

| 候補 | 採否 | 評価 |
| --- | --- | --- |
| TypeScript + Web Worker | **採用** | 決定性・型契約・AI 協働の三拍子。性能は §1.5.2 のホットスポット最適化で対応 |
| Rust + WASM（全面） | 不採用 | 性能は最高だが、ビルド工程の追加と AI 協働の鈍化、Pasta 仕様変更時の追従コスト |
| UI スレッドで同期実行 | 不採用 | パラメータ変更で UI が固まり、対話性が損なわれる |
| サーバ側計算 | 不採用 | [要件定義書.md §3.4](要件定義書.md) のオフライン要件に矛盾 |

#### 1.5.1 純粋関数の規約

各 Step は `step(input: TIn): TOut` の形を取る。`TIn` には先行 Step の出力と惑星パラメータを含み、副作用・非決定性（乱数の種子なし呼び出し・現在時刻参照など）を許さない。乱数を使う Step は種子を `TIn` に含める。

#### 1.5.2 ホットスポット最適化の方針

性能ボトルネックが明らかになった Step について、Rust + WASM 化する将来オプションを許容する。判断は実測ベース（[要件定義書.md §3.1](要件定義書.md) の数十秒オーダー要件を満たせない場合）で行い、本仕様策定時点では実施しない。

### 1.6 描画

**選定**: マップは Canvas 2D を一次。補助オーバーレイ（凡例・補助線・対話要素）は SVG / React コンポーネント。グラフは uPlot。WebGL は性能要件を満たせない場合のホットスポット限定オプション。

**理由**:

- 緯度経度グリッドのラスター表示は Canvas 2D が最適（DOM ノード爆発を回避）。
- 補助線・凡例・ツールチップは SVG / React コンポーネントで宣言的に記述でき、対話処理が容易。
- グラフ（緯度プロファイル・季節タイムライン）は uPlot で軽量に描画。Recharts / D3 は React と統合に独自パターンが必要で旧版で問題化した。
- 旧版の D3 + Recharts は React の宣言的レンダリングと命令的 DOM 操作の不整合を生じた。本仕様では Canvas（命令的）を React 外で扱い、SVG（宣言的）を React 内で扱うことで分離する。

**候補比較**:

| 候補 | 採否 | 評価 |
| --- | --- | --- |
| Canvas 2D 一次 + SVG オーバーレイ + uPlot | **採用** | 旧版の D3+Recharts 問題を回避しつつ性能・対話性を両立 |
| SVG 全面 | 不採用 | 高解像度マップで DOM ノードが膨大になり性能劣化 |
| WebGL 全面 | 不採用 | 開発コストと AI 協働の鈍化が大きい。ホットスポットのみで使う |
| D3 + Recharts（旧版同等） | 不採用 | React との統合パターンが煩雑で旧版と同じ問題を踏む |

### 1.7 ビルドツール

**選定**: Vite 6.x

**理由**:

- §1.2 TypeScript と §1.3 React に最良適合。
- HMR が高速で、AI 協働の編集 → 確認サイクルが速い。
- 静的 SPA としてのビルド出力が単純。

**候補比較**:

| 候補 | 採否 | 評価 |
| --- | --- | --- |
| Vite 6.x | **採用** | 標準・高速・設定単純 |
| Webpack | 不採用 | 設定が複雑、HMR が遅い |
| Rspack / Turbopack | 不採用 | 新興で安定性懸念、Vite に明確な優位がない |

### 1.8 テスト

**選定**:

- 単体・契約テスト: Vitest
- E2E: Playwright
- 数値検証: 専用ハーネス（地球パラメータ再現性 / Pasta 公開図比較 / 仮想大陸）

**理由**:

- Vitest は Vite 統合・TypeScript ネイティブで、§1.5 の純粋関数 Step を型レベルで検証できる。
- Playwright はクロスブラウザ E2E に優れ、Tauri 化時にも流用可能。
- 数値検証は [要件定義書.md §2.5](要件定義書.md) の検証要件を直接満たすために専用ロジックが必要。

**候補比較**:

| 候補 | 採否 | 評価 |
| --- | --- | --- |
| Vitest + Playwright | **採用** | Vite 統合・クロスブラウザ・型ネイティブ |
| Jest | 不採用 | Vite との設定相性が微妙、TypeScript ネイティブでない |
| Cypress | 不採用 | Playwright より遅く、複数タブ等の制約あり |

## 2. アーキテクチャ

### 2.1 レイヤー分割

ExoClim は以下の 5 層に分割する。各層の責務を明確化し、層間の依存方向を上から下への一方向に固定する。

```
[ UI 層（React コンポーネント） ]
        ↓ 参照
[ 状態層（Zustand store） ]
        ↓ 参照
[ ワーカー層（Web Worker 上の pipeline 呼び出し） ]
        ↓ 参照
[ 計算層（純粋関数の Step 群） ]
        ↓ 参照
[ ドメイン層（型定義・物理定数・地形データ） ]
```

#### 2.1.1 ドメイン層

- 責務: 型定義（PlanetParams, Grid, StepInput, StepOutput）、物理定数（[docs/spec/](../docs/spec/) で参照される定数）、地形データの読み込み。
- 規約: 副作用を持たない。計算ロジックは含まない。
- ディレクトリ: `src/domain/`

#### 2.1.2 計算層

- 責務: 各 Step（ITCZ・風帯・海流・気流・気温・降水・気候帯）の物理計算を純粋関数として実装。
- 規約:
    - `step(input: TIn): TOut` 形式に統一（[§1.5.1](#151-純粋関数の規約)）。
    - 副作用なし。乱数は `input` に種子を含める。
    - 各 Step は他 Step の実装に依存しない（型のみ参照）。
    - 各 Step ファイルの先頭に対応する [docs/spec/0X_*.md](../docs/spec/) の節番号を引用コメントで明記する。
- ディレクトリ: `src/sim/`（Step ごとに 1 ファイル）

#### 2.1.3 ワーカー層

- 責務: 計算層の pipeline 全体を Web Worker 上で順に実行し、結果を状態層に返す。部分再計算（変更パラメータ以降の Step のみ）を担う。
- 規約: ワーカー本体は計算層に依存できるが UI 層・状態層には依存しない。
- ディレクトリ: `src/worker/`

#### 2.1.4 状態層

- 責務: Zustand store によるパラメータ・結果・UI 状態の保持。ワーカーへの計算依頼と結果受信のオーケストレーション。
- 規約: 計算層の純粋関数を直接呼ばない（必ずワーカー経由）。store は責務別に複数に分割する。
- ディレクトリ: `src/store/`

#### 2.1.5 UI 層

- 責務: React コンポーネント。マップ表示、グラフ、パラメータ調整 UI、デバッグビュー。
- 規約: 状態層の store のみを参照する。計算層・ワーカー層への直接依存を禁止する。
- ディレクトリ: `src/ui/`（機能別に再帰的にディレクトリ分割）

### 2.2 ステップ間の値渡し

#### 2.2.1 入出力契約

各 Step は TypeScript の型として `TIn` / `TOut` を持つ。型は `src/domain/` に集約し、計算層と状態層の双方が参照する。

旧版の問題（風帯派生 gap `oceanEcLatGapDerived` が UI に伝搬しない）を防ぐため、Step の出力に含まれるすべての値は、UI 層から参照可能な store にも反映できる構造とする。**「内部派生値で UI から見えない量」を許さない**（[要件定義書.md §2.3.6](要件定義書.md) の「UI 未露出のパラメータは存在しない」要件と整合）。

#### 2.2.2 段階間フィードバック

[要件定義書.md §2.2.8](要件定義書.md) で許容したフィードバック経路（ITCZ↔気温・海流↔気温・気流↔気温・雪氷自己強化）は、最大反復回数 N（既定 N = 3）の固定ループとして実装する。N は §3 利便性の応答性要件と矛盾しない範囲で利用者調整可能とする。

各反復で結果が収束したか（前回反復との差分がしきい値以下）を判定し、収束時は早期終了する。

#### 2.2.3 部分再計算

ワーカー層は、変更パラメータが影響する最も早い Step を特定し、その Step 以降のみを再計算する。各 Step は前回の出力をキャッシュし、入力が同一なら再実行をスキップする。

キャッシュキーは `TIn` の構造的同値性（deep equality）で判定する。乱数種子・浮動小数点数の比較は厳密一致とする（決定性要件、[要件定義書.md §3.2](要件定義書.md)）。

### 2.3 検証ハーネス

#### 2.3.1 ヘッドレス実行

Node.js 上で計算層を直接呼び出し、Step の出力をファイル（PNG / JSON）として保存する。Web Worker は使わない（決定性確保および差分比較容易性のため）。

[要件定義書.md §2.5](要件定義書.md) の検証要件を、以下の三段階のハーネスとして実装する。

- **地球パラメータ再現性**: `tests/fixtures/earth.json` を入力し、出力マップを地球の Köppen-Geiger 公開データと比較。
- **Pasta 公開図比較**: Pasta が「An Apple Pie from Scratch」で示す例題（地球・特殊軌道）を入力し、出力を Pasta 公開図と並置する HTML レポートを生成。
- **仮想大陸**: `tests/fixtures/idealized_continent_*.json` を入力し、典型的な気候配置（西岸海洋性気候・内陸乾燥帯）の出現を検証。

#### 2.3.2 検証フィクスチャの管理

検証用フィクスチャ（地球パラメータ・理想化大陸）は `tests/fixtures/` に置き、版管理する。Pasta 公開図そのものは著作権の関係でリポジトリに含めず、利用者が一次資料 URL から手元に取得して `tests/fixtures-external/` に配置する運用とする。

旧 ExoClim から引き継ぐ理想化大陸画像（[現状.md M2](現状.md)）は、`tests/fixtures-external/` 側に配置する。

#### 2.3.3 比較レポート

ヘッドレスハーネスは比較対象が手元にある場合のみ並置レポートを出力し、欠落時はその旨を明示してテスト自体は通す（外部資産の有無で CI が壊れないようにする）。

## 3. 設計判断の根拠

各選定が [要件定義書.md](要件定義書.md) のどの要件に応えているかの対応表。

| 選定 | 要件定義書の対応箇所 | 旧版からの変化 |
| --- | --- | --- |
| §1.1 静的 Web SPA | §3.4 オフライン動作・配布 | 同等（Web SPA） |
| §1.2 TypeScript | §3.3 仕様と実装の対応 | 同等 |
| §1.3 React | （プラットフォーム要件） | 同等 |
| §1.4 Zustand | §3.1 部分再計算 | Context 中心から Zustand に変更 |
| §1.5 TypeScript + Worker | §3.1 計算時間・部分再計算、§3.2 決定性 | UI スレッド実行から Worker に変更 |
| §1.6 Canvas + SVG + uPlot | §2.3 表示・操作 | D3+Recharts から変更 |
| §1.7 Vite | （ビルド要件） | 同等 |
| §1.8 Vitest + Playwright + 数値ハーネス | §2.5 検証 | E2E と数値検証の体系化 |
| §2.1 5 層分割 | §3.3 保守性 | レイヤー間依存を一方向に固定 |
| §2.2.1 入出力契約 | §2.3.6「UI 未露出パラメータなし」 | 旧版の派生値伝搬問題への直接対応 |
| §2.2.2 反復フィードバック | §2.2.8 段階間フィードバック | 単方向計算からの拡張 |
| §2.2.3 部分再計算 | §3.1 性能 | キャッシュ機構の明示 |
| §2.3 検証ハーネス | §2.5 検証 | ヘッドレス実行と並置レポートの体系化 |

旧版から変えていない選定（TypeScript・Vite・React）は、旧版の問題が platform に起因していなかったためである。governance / design レベルでの改善（§1.4・§1.5・§1.6・§2.1・§2.2）が本仕様の主な焦点である。
